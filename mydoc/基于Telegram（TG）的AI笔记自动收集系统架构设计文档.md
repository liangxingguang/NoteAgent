# 基于Telegram（TG）的AI笔记自动收集系统架构设计文档

# 1\. 文档概述

## 1\.1 文档目的

本文档旨在明确基于Telegram（TG）机器人的AI笔记自动收集系统的整体架构设计，涵盖架构分层、组件职责、数据流向、部署架构及约束限制，为系统开发、部署及维护提供标准化依据。本架构核心实现「手机端TG消息/文件收集→第三方AI总结→Obsidian自动入库」的全自动化链路，适配国内用户（需科学上网），兼顾稳定性、易用性及可扩展性。

## 1\.2 核心需求

- 消息收集：支持TG机器人接收用户发送的纯文本、链接、PDF/Word/TXT文件，实现多类型内容采集。

- AI处理：调用第三方大模型（通义千问/GPT\-3\.5等），对收集的文本/文件解析内容进行标准化总结，生成Obsidian兼容格式笔记。

- 自动入库：将AI总结后的笔记自动写入服务器本地Obsidian知识库，支持多设备同步。

- 稳定运行：实现机器人7×24小时后台常驻，支持断线自动重连，具备权限控制防滥用。

- 低门槛部署：服务器配置要求低，部署流程简洁，支持Linux系统后台常驻。

## 1\.3 架构范围

本架构覆盖从用户消息发送、TG机器人接收，到AI总结、Obsidian入库的全链路，不包含第三方大模型本身的部署、Obsidian客户端的安装及多设备同步的具体配置（仅提供同步方案建议），也不涉及TG账号注册、API Key申请的具体操作流程。

# 2\. 整体架构设计

## 2\.1 架构分层（自上而下）

系统采用分层架构设计，共分为5层，各层职责清晰、松耦合，便于维护和扩展，整体架构如下：用户交互层 → 接入层 → 业务逻辑层 → 数据存储层 → 基础设施层。

### 2\.1\.1 用户交互层

核心职责：提供用户与系统的交互入口，实现内容收集的便捷操作。

- 交互载体：Telegram客户端（手机端/PC端），用户通过私聊方式向TG机器人发送文本、链接、文件。

- 交互操作：用户发送内容（文本/链接/文件）、接收机器人处理结果反馈（入库成功提示、失败原因）。

- 约束：用户需具备TG账号，且能正常访问TG（国内用户需科学上网）；仅允许指定用户使用，防止滥用。

### 2\.1\.2 接入层

核心职责：负责与Telegram官方API对接，实现消息接收、发送及机器人身份认证，是系统与TG平台的桥梁。

- 核心组件：TG机器人接入模块、消息解析模块、权限校验模块。

- 核心能力：
       

    - 机器人认证：通过TG BotFather创建的机器人Token，与TG官方API建立连接，完成身份校验。

    - 消息接收：采用Polling模式（无需公网IP/域名），实时监听用户发送的消息，区分文本、链接、文件类型。

    - 消息解析：提取文本内容、链接信息；对文件进行初步校验（大小、格式），并触发下载逻辑。

    - 权限校验：校验发送消息的用户ID，仅允许预设的合法用户使用系统，拒绝非法请求。

    - 消息反馈：向用户发送处理结果（成功/失败提示、入库文件名）。

### 2\.1\.3 业务逻辑层

核心职责：实现系统核心业务逻辑，串联接入层、数据存储层及第三方服务，是系统的核心处理中心。

- 核心组件：文件处理模块、AI总结模块、Obsidian入库模块、异常处理模块。

- 核心能力：
        

    - 文件处理：下载用户发送的文件（PDF/Word/TXT），提取文件中的文本内容，处理完成后清理临时文件，避免服务器空间占用。

    - AI总结：调用第三方大模型API（通义千问/GPT\-3\.5等），将文本/文件提取内容传入模型，按照预设模板生成Obsidian标准化笔记（标题、摘要、关键词、正文）。

    - Obsidian入库：生成唯一文件名（时间戳\+内容哈希），将AI总结后的笔记写入服务器本地Obsidian知识库目录，确保文件不重复、可追溯。

    - 异常处理：捕获消息接收、文件下载、AI调用、入库过程中的异常（如文件过大、AI调用失败、权限不足），记录日志并向用户反馈失败原因。

### 2\.1\.4 数据存储层

核心职责：负责系统数据的持久化存储，包括AI总结后的笔记、系统日志及临时文件。

- 核心存储载体：
        

    - Obsidian知识库：存储AI总结后的Markdown格式笔记，目录结构由用户预设，支持Obsidian客户端直接访问及多设备同步（如通过Git插件）。

    - 临时文件目录：存储下载的用户文件，处理完成后自动删除，仅用于临时解析。

    - 日志文件：记录系统运行状态、消息处理记录、异常信息，用于问题排查和维护。

- 存储约束：Obsidian目录需具备可读写权限；临时文件目录需定期清理（或自动清理），避免空间溢出；日志文件可设置滚动存储，限制单个日志文件大小。

### 2\.1\.5 基础设施层

核心职责：为系统提供运行环境和基础支撑，保障系统稳定运行。

- 运行环境：Linux服务器（推荐Ubuntu 22\.04/Debian 12），Python 3\.8\+运行环境。

- 网络支撑：服务器需能访问TG官方API及第三方大模型API（国内服务器需科学上网）；支持Outbound网络连接，无需开放公网IP/端口。

- 后台常驻：通过Linux systemd服务实现机器人进程的开机自启、断线自动重连，确保7×24小时运行。

- 第三方服务：第三方大模型API（通义千问/GPT\-3\.5等），提供AI总结能力；TG官方API，提供机器人接入能力。

## 2\.2 组件交互关系

各层组件通过接口调用、数据传递实现协同工作，核心交互流程如下：

1. 用户通过TG客户端向机器人发送文本/链接/文件，触发交互请求。

2. 接入层的TG机器人接入模块接收请求，通过权限校验模块校验用户合法性。

3. 合法请求由消息解析模块解析消息类型，文本/链接直接传递至业务逻辑层；文件则触发下载逻辑，下载至临时目录后传递至业务逻辑层。

4. 业务逻辑层的文件处理模块提取文件文本，与文本/链接内容一同传入AI总结模块，调用第三方大模型生成标准化笔记。

5. AI总结完成后，Obsidian入库模块生成唯一文件名，将笔记写入Obsidian知识库。

6. 业务逻辑层将处理结果反馈至接入层，接入层通过TG机器人向用户发送反馈消息。

7. 异常处理模块全程捕获异常，记录日志并向用户反馈失败原因，同时清理临时资源。

# 3\. 核心流程设计

## 3\.1 整体业务流程

系统核心业务流程为「用户发送内容→机器人接收→AI总结→入库反馈」，全程自动化，无需人工干预，具体流程如下：

1. 用户操作：打开TG客户端，向指定TG机器人发送文本、链接或PDF/Word/TXT文件。

2. 消息接收：TG机器人接入模块实时监听消息，完成权限校验和消息解析。

3. 内容处理：文本/链接直接进入AI总结环节；文件先下载、提取文本，再进入AI总结环节。

4. AI总结：调用第三方大模型，按照预设模板生成Obsidian笔记（标题、摘要、关键词、正文）。

5. 自动入库：将生成的Markdown笔记写入Obsidian知识库，生成唯一文件名。

6. 结果反馈：机器人向用户发送处理结果，告知入库成功及文件名；若失败，告知失败原因。

7. 资源清理：清理临时文件，记录系统日志。

## 3\.2 关键子流程设计

### 3\.2\.1 文件处理子流程

1. 接入层解析到文件类型消息，获取文件名称、大小、格式等信息。

2. 校验文件大小（默认≤50MB，官方Bot API限制）和格式（仅支持PDF/Word/TXT）。

3. 若文件符合要求，调用TG API下载文件至服务器临时目录。

4. 文件处理模块根据文件格式，提取文本内容（PDF用PDF解析库、Word用Word解析库、TXT直接读取）。

5. 文本内容过长时，自动截断（避免AI API请求超限），传递至AI总结模块。

6. 处理完成后，删除临时文件，释放服务器空间。

### 3\.2\.2 AI总结子流程

1. 业务逻辑层接收文本内容（文本/链接/文件提取结果），构造AI总结提示词（含Obsidian笔记模板）。

2. 调用第三方大模型API，传入提示词和文本内容，设置超时时间（避免请求阻塞）。

3. 接收AI返回的总结结果，校验结果格式是否符合要求。

4. 若格式符合，传递至Obsidian入库模块；若不符合或调用失败，记录异常日志，向用户反馈失败原因。

### 3\.2\.3 断线重连子流程

1. 基础设施层的systemd服务实时监控机器人进程状态。

2. 若机器人进程异常退出（如网络波动、API调用失败），systemd服务在5秒内自动重启进程。

3. 进程重启后，接入层重新与TG官方API建立连接，完成机器人登录，恢复消息监听。

4. 重启成功后，记录日志；若多次重启失败，记录异常并触发告警（可选）。

# 5\. 约束与限制

- 消息/文件限制：官方Bot API默认支持单文件≤50MB，机器人下载用户文件≤20MB；若需支持更大文件，需部署本地Bot API服务器（可扩展至2GB）。

- 风控限制：TG对第三方机器人有一定风控，若机器人存在异常行为（如频繁发送垃圾消息、异地登录），可能被限制或封号，需规范使用。

- 网络限制：国内用户需科学上网才能使用TG，服务器需具备访问TG的网络能力。

## 5\.2 第三方大模型约束

- API额度：第三方大模型API通常有免费额度（如通义千问qwen\-turbo免费额度足够个人使用），免费额度用完后需付费。

- 内容限制：AI模型对敏感内容有过滤机制，若用户发送敏感内容，AI可能拒绝总结，需规范发送内容。

## 5\.3 系统自身约束

- 文件格式限制：仅支持PDF/Word/TXT文件的文本提取，不支持图片、视频、压缩包等其他格式。

- 文本长度限制：文件提取的文本超过5000字将自动截断，避免AI API请求超限。

- 权限约束：仅允许预设的合法用户使用，防止陌生人滥用机器人消耗API额度。
