# LLM Wiki 功能详细设计文档（按日归档版）

## 一、文档概述

### 1.1 文档目的

本文档基于 NoteAgents 项目现有架构，设计 LLM Wiki 功能，采用**按日归档**策略，实现：

- **双向统一加工**：TG+飞书双平台自动收集内容 + 人工手写笔记，共用一套 LLM 加工逻辑
- **三层目录结构**：原始存档（raw/）→ LLM 结构化精加工（structured/）→ 全局索引（index.md）
- **按日归档管理**：原始内容按"年/月/日"精细化归档，便于精确追溯
- **全自动处理**：后台异步执行，不阻塞 NoteAgents 主流程

### 1.2 核心设计思想

**Karpathy 观点**："No database. No vector embeddings. No server. Just files and a capable model."

**豆包优化**：兼容 TG+飞书双平台自动收集 + 人工手写笔记双场景，统一 LLM 加工逻辑，实现"保存即自动整理"。

### 1.3 按日归档说明

原始存档层（raw/）采用**按日归档**策略：

- **归档粒度**：每条笔记按"年/月/日"三级目录结构归档
- **目录格式**：`raw/auto/2026/05/03/` 或 `raw/manual/2026/05/03/`
- **优势**：可精确追溯某一天的所有笔记，定位精准，适合笔记量大的场景
- **影响评估**：目录数量增加（相比按月归档约30倍），但单日笔记量通常有限，实际影响可控

## 二、架构设计

### 2.1 整体架构（五层架构）

| 层级 | 名称 | 职责 |
|------|------|------|
| 1 | 交互入口层 | TG Bot、Feishu Bot 自动收集入口 + 人工手动编辑入口 |
| 2 | 原始存档层（raw/） | 按日归档存储，包含 raw/auto/ 和 raw/manual/ 两个子目录 |
| 3 | LLM结构化精加工层（structured/） | 按内容类型分类：技术类、想法类、学习类、日常类 |
| 4 | Wiki知识网络层（wiki/） | 实体、概念、对比分析、查询结果 |
| 5 | 全局索引层（index.md） | 自动更新内容索引和统计信息 |

### 2.2 目录结构（按日归档）

```
wiki/
├── SCHEMA.md                    # 规则定义文档（Agent操作手册）
├── index.md                     # 全局索引文档
├── log.md                       # 操作日志文档（仅追加）
├── raw/                         # 原始存档目录（按日归档）
│   ├── auto/                    # 双平台自动收集内容
│   │   ├── 2026/                # 年份目录
│   │   │   └── 05/              # 月份目录
│   │   │       └── 03/          # 日目录（按日归档）
│   │   │           └── *.md     # 当日原始笔记
│   │   └── ...
│   └── manual/                  # 人工手写笔记
│       ├── 2026/
│       │   └── 05/
│       │       └── 03/
│       │           └── *.md
├── structured/                  # LLM结构化精加工目录（按内容类型）
│   ├── 技术类/                  # 技术相关笔记
│   ├── 想法类/                  # 想法/观点相关笔记
│   ├── 学习类/                  # 学习心得相关笔记
│   └── 日常类/                  # 日常生活相关笔记
└── wiki/                        # Wiki知识网络
    ├── entities/                # 实体页目录
    ├── concepts/                # 概念页目录
    ├── comparisons/             # 对比分析目录
    └── queries/                 # 查询结果目录
```

## 三、核心处理流程

### 3.1 统一处理流程

所有内容（TG+飞书双平台自动收集、人工手写）均遵循统一处理流程：

```
内容输入 → 原始存档（按日归档） → LLM结构化加工 → 分类保存 → 索引更新
```

**步骤详解**：

1. **内容输入**
   - 来源一：TG+飞书双平台自动收集的内容
   - 来源二：人工手写的笔记

2. **原始存档（按日归档）**
   - 自动收集内容：保存至 `raw/auto/{年}/{月}/{日}/`
   - 人工手写笔记：保存至 `raw/manual/{年}/{月}/{日}/`
   - 示例：`raw/auto/2026/05/03/note_001.md`
   - 所有原始内容永久保留、不做修改

3. **LLM 结构化加工**
   - 提炼核心标题
   - 精简摘要（80字以内）
   - 提取关键词/标签（3-5个）
   - 梳理核心要点（3-5条）
   - 添加 Obsidian 双向链接
   - 自动分类（技术/想法/学习/日常）

4. **分类保存**
   - 加工后的结构化内容保存至 `structured/{类型}/`

5. **索引更新**
   - 自动更新 `index.md`，同步新增内容的导航信息

### 3.2 人工笔记自动处理流程

```
用户保存笔记 → 文件监控检测 → 读取内容 → LLM加工 → 分类保存 → 索引更新
```

1. 用户在 `raw/manual/{年}/{月}/{日}/` 目录下保存 `.md` 格式笔记
2. 后台异步文件监控器实时检测目录变化
3. 监控器自动读取笔记内容，提交至 LLM 加工模块
4. LLM 加工模块按照统一规则进行结构化处理
5. 加工完成后，保存至 `structured/{类型}/` 对应目录
6. 自动触发全局索引更新

## 四、LLM 处理模块设计

### 4.1 核心定位

LLM 处理模块是 LLM Wiki 功能的核心，负责所有原始内容的结构化加工，实现内容的标准化、简洁化和关联化。

### 4.2 统一加工规则

| 任务 | 说明 |
|------|------|
| 提炼核心标题 | 基于原始内容，提炼简洁、准确、贴合主题的标题 |
| 精简摘要 | 将核心信息浓缩至80字以内 |
| 提取关键词/标签 | 提取3-5个核心关键词/标签，用于分类和检索 |
| 梳理核心要点 | 将核心信息拆解为3-5条要点 |
| 添加双向链接 | 识别核心概念、关键词，添加 Obsidian 双向链接 |
| 自动分类 | 自动划分为技术类、想法类、学习类、日常类 |
| 内容优化 | 剔除冗余、口语化信息，保持逻辑清晰、忠于原文 |

### 4.3 核心能力

- **双来源适配**：同时支持 auto 类型和 manual 类型
- **多模型兼容**：支持多种 LLM 模型，可通过配置切换
- **重试机制**：API 调用失败时自动重试
- **标准化输出**：统一格式规范，便于索引和知识网络构建

## 五、文件监控模块设计

### 5.1 核心定位

文件监控模块是实现人工笔记"保存即自动整理"的关键，采用轻量化设计、无第三方依赖，以后台异步线程形式运行。

### 5.2 核心功能

| 功能 | 说明 |
|------|------|
| 实时监控 | 持续监控 `raw/manual/` 目录，捕捉新文件创建、文件修改事件 |
| 异步处理 | 后台线程运行，不占用主流程资源 |
| 自动触发加工 | 检测到变化后，自动提交至 LLM 加工模块 |
| 异常处理 | 跳过异常文件，记录日志，继续监控 |
| 可配置性 | 监控间隔可通过配置文件调整 |

### 5.3 运行机制

1. 模块启动后，扫描 `raw/manual/` 目录下已有的所有 `.md` 文件
2. 按设定间隔定期扫描目录，对比文件列表变化
3. 对新增和修改的文件，逐一进行读取和加工处理
4. 处理完成后更新已监控文件列表，循环执行

## 六、Wiki 知识网络（Karpathy 范式）

### 6.1 核心操作

| 操作 | 说明 |
|------|------|
| **导入（Ingest）** | 将结构化内容导入 Wiki，自动提取实体、概念，建立双向链接 |
| **查询（Query）** | 基于全局索引检索相关内容，LLM 综合生成查询结果 |
| **健康检查（Lint）** | 定期检查孤立页面、断裂链接、过时内容 |

### 6.2 标准化页面类型

| 类型 | 说明 |
|------|------|
| **实体页** | 记录人、组织、产品等实体，包含一句话总结、详细信息、相关概念、来源 |
| **概念页** | 记录技术、认知等概念，包含一句话总结、详细说明、相关实体、来源 |

## 十、按日归档 vs 按月归档对比

| 对比项 | 按日归档 | 按月归档 |
|--------|----------|----------|
| 目录数量 | 约365个/年 | 约12个/年 |
| 定位精度 | 精确到"年月日" | 精确到"年月" |
| 适用场景 | 笔记量大、需细粒度管理 | 笔记量适中、目录简洁优先 |
| 查找效率 | 快速定位某日内容 | 需结合文件名搜索 |
| 管理复杂度 | 略高（目录更多） | 较低（目录更少） |

**推荐**：笔记量大（每日10条以上）建议按日归档；笔记量适中建议按月归档。系统支持通过配置灵活切换归档策略。

## 十一、关键设计要点

1. **100% 自研**：无 LangChain、无 Agent 框架，纯 Python 实现
2. **分层隔离**：原始文件永久保留、结构化内容独立存储、Wiki 单独沉淀
3. **全自动化**：后台异步执行，不阻塞主流程
4. **按日归档**：原始存档按"年/月/日"精细化归档，便于精确追溯
5. **统一加工**：双来源（自动/手动）采用同一套 LLM 加工逻辑
