# LLM Wiki 功能详细设计文档（优化版）

注意：文档中初始配置的通义千问兼容模式链接（https://dashscope\.aliyuncs\.com/compatible\-mode/v1）及（https://dashscope\.aliyuncs\.com/compatible\-mode/v1/chat/completions）均存在网页解析失败问题，推测为链接格式不完整、访问权限异常或模型配置不匹配，后续需替换为可正常访问的 LLM API 地址（如其他兼容 OpenAI 格式的模型链接），或检查通义千问 API 权限与格式配置。

# 1\. 文档概述

## 1\.1 文档目的

本文档基于 NoteAgents 项目现有架构，设计 LLM Wiki 功能，整合 Karpathy 标准范式与豆包优化思路，实现：

- **双向统一加工**: TG\+飞书双平台自动收集内容 \+ 人工手写笔记，共用一套 LLM 加工逻辑，无需分开配置，实现内容处理标准化

- **三层目录结构**: 原始存档（raw/）\+ LLM 结构化精加工（structured/）\+ 全局索引（index\.md），分层隔离、可追溯，保障数据安全

- **全自动处理**: 自动识别内容类型、智能分类、标签关联、去重，后台异步执行，不阻塞 NoteAgents 主流程

## 1\.2 核心思想

**Karpathy 观点**: \&\#34;No database\. No vector embeddings\. No server\. Just files and a capable model\.\&\#34;（无需数据库、向量嵌入、服务器，仅通过文件和高性能模型实现轻量、高效的知识库）

**豆包优化**: 兼容 TG\+飞书双平台自动收集 \+ 人工手写笔记双场景，统一 LLM 加工逻辑，实现\&\#34;保存即自动整理\&\#34;，解决纯 Karpathy 方案手动导入、处理低效的痛点，适配自研 Agent 无框架架构，降低使用门槛。

## 1\.3 设计原则

1. **100% 自研**: 无 LangChain、无 Agent 框架，纯 Python 实现，降低第三方依赖，提升代码可扩展性、可控性和可维护性

2. **双兼容**: TG\+飞书双平台自动收集内容与人工手写笔记，采用同一套 LLM 加工、分类、索引逻辑，避免流程割裂，保障内容处理一致性

3. **分层隔离**: 原始文件（raw/）永久保留、不修改，结构化内容（structured/）独立存储，Wiki 知识网络（wiki/）单独沉淀，确保数据可追溯、不丢失，同时避免不同层级内容相互干扰

4. **全自动化**: 所有内容处理流程均为后台异步执行，不阻塞 NoteAgents 主流程（如 TG/飞书机器人消息接收、任务调度等），实现\&\#34;无需手动干预、保存即整理\&\#34;的高效体验

# 2\. 架构设计

## 2\.1 整体架构

整体分为五层架构，从交互入口到全局索引，形成完整的内容流转、加工、沉淀链路，各层级职责清晰、衔接顺畅，适配双内容来源（TG\+飞书双平台自动收集、人工手写）的核心需求：

5. **交互入口层**: 负责内容的输入，包括 Telegram Bot、Feishu Bot 两种自动收集入口，以及人工手动编辑入口，覆盖不同使用场景，适配系统已支持的双平台收集需求

6. **原始存档层（raw/）**: 作为内容的\&\#34;原始备份区\&\#34;，分为双平台自动收集（raw/auto/）和人工手写（raw/manual/）两个子目录，所有原始内容均在此层永久保留，不做任何修改，用于后续追溯和二次加工

**LLM 结构化精加工层（structured/）**: 对原始内容进行标准化加工，按照内容类型自动划分为技术类、想法类、学习类、日常类四个默认子目录，支持子目录的新增、删除操作，同时支持已分类内容在不同子目录间的迁移，输出结构化、简洁化、可直接使用的笔记内容

7. **Wiki 知识网络层（wiki/）**: 基于结构化内容，提炼实体、概念、对比分析和查询结果，形成相互关联的知识网络，实现知识的深度沉淀和关联检索，贴合 Karpathy 标准范式

8. **全局索引层（index\.md）**: 作为知识库的\&\#34;导航目录\&\#34;，自动更新所有层级的内容索引，方便快速定位、检索各类内容，同时展示知识库统计信息，实现全局管控

## 2\.2 目录结构

采用标准化目录结构，与 Obsidian 知识库无缝兼容，各目录职责明确、层级清晰，便于后续维护和扩展，具体结构如下：

- **wiki/**: LLM Wiki 根目录，与 Obsidian 共用，统一管理所有知识库相关内容

- **wiki/SCHEMA\.md**: 规则定义文档，作为 Agent 操作手册，明确内容加工、分类、索引的统一规则，确保所有处理流程标准化

- **wiki/index\.md**: 全局索引文档，自动更新，展示所有层级内容的导航和统计信息

- **wiki/log\.md**: 操作日志文档，仅追加不修改，记录所有内容加工、索引更新、监控触发等操作，用于问题排查和流程追溯

- **wiki/raw/**: 原始存档目录，永久保留所有原始内容，按来源分为两个子目录，均按月分目录存储，便于管理

- **wiki/raw/auto/**: 存储 TG\+飞书双平台自动收集的内容，按月分目录归档

- **wiki/raw/manual/**: 存储人工手写的笔记，按月分目录归档

- **wiki/structured/**: LLM 结构化精加工目录，默认按内容类型分为技术类、想法类、学习类、日常类四个子目录，支持子目录的新增、删除，以及已归档内容在各子目录间的迁移，存储加工后的标准化笔记

- **wiki/wiki/**: Wiki 知识网络目录，贴合 Karpathy 范式，按知识类型分为四个子目录，实现知识的深度沉淀和关联

- **wiki/wiki/entities/**: 实体页目录，存储人、组织、产品等实体相关的标准化页面

- **wiki/wiki/concepts/**: 概念页目录，存储各类技术、认知等概念相关的标准化页面

- **wiki/wiki/comparisons/**: 对比分析目录，存储不同实体、概念的对比分析内容

- **wiki/wiki/queries/**: 查询结果目录，存储基于知识库的各类查询结果，便于后续复用

# 3\. 核心处理流程

## 3\.1 内容处理流程

所有内容（TG\+飞书双平台自动收集、人工手写）均遵循统一的处理流程，确保加工逻辑一致、输出格式统一，整体流程分为五个核心步骤：

9. **内容输入**: 分为两个来源，一是 TG\+飞书双平台自动收集的内容，二是人工手写的笔记，两个来源的内容分别进入对应原始存档目录

10. **原始存档**: TG\+飞书双平台自动收集的内容保存至 raw/auto/ 目录，人工手写笔记保存至 raw/manual/ 目录，均按月份归档，永久保留、不做修改

11. **LLM 结构化加工**: 对两个来源的原始内容，采用同一套 LLM 加工逻辑，进行标题提炼、摘要精简、关键词提取、核心要点梳理、双向链接添加和内容分类

12. **分类保存**: 加工后的结构化内容，根据自动划分的类型（技术类、想法类、学习类、日常类），保存至 structured/ 对应默认子目录；支持对该目录下的子目录进行新增、删除操作，同时支持已保存的结构化内容在不同子目录间迁移，灵活适配内容分类需求

13. **索引更新**: 内容保存完成后，自动更新全局索引（index\.md），同步新增内容的导航信息，确保索引与实际内容一致

## 3\.2 人工笔记自动处理流程

针对人工手写笔记，设计专属自动处理流程，实现\&\#34;保存即整理\&\#34;，无需手动触发加工，流程如下：

1. 用户在 raw/manual/ 目录下保存 \.md 格式的手写笔记

2. 后台异步文件监控器（轻量化、无第三方依赖）实时检测目录变化，捕捉到新文件创建或现有文件修改事件

3. 监控器自动读取该手写笔记的内容，提交至 LLM 加工模块进行结构化处理

4. LLM 加工模块按照统一规则，对笔记内容进行标准化加工，生成结构化内容并自动划分分类

5. 加工完成后，将结构化内容保存至 structured/ 对应分类目录

6. 自动触发全局索引更新，将该笔记的导航信息添加至 index\.md，完成整个处理流程

# 4\. LLM 处理模块设计

## 4\.1 核心定位

LLM 处理模块是整个 LLM Wiki 功能的核心，负责所有原始内容的结构化加工，实现内容的标准化、简洁化和关联化，适配双内容来源，同时兼容多种 LLM 模型，可通过配置快速切换，无需修改核心逻辑。

## 4\.2 统一加工规则

采用统一的加工规则（Prompt 逻辑），确保所有内容输出格式一致、信息完整，核心加工任务包括：

1. 提炼核心标题：基于原始内容，提炼简洁、准确、贴合主题的标题，便于检索和识别

2. 精简摘要：将原始内容核心信息浓缩至80字以内，快速呈现内容核心，提升阅读效率

3. 提取关键词/标签：提取内容中的核心关键词，生成3\-5个标签，用于分类和检索

4. 梳理核心要点：将原始内容的核心信息拆解为3\-5条要点，逻辑清晰、重点突出

5. 添加双向链接：识别内容中的核心概念、关键词，添加 Obsidian 双向链接，为知识网络构建奠定基础

6. 自动分类：根据内容主题，自动划分为技术类、想法类、学习类、日常类四类，确保分类统一

7. 内容优化：剔除原始内容中的冗余、口语化信息，保持内容逻辑清晰、语言简洁，同时忠于原文，不编造、不扩写信息

## 4\.3 核心能力

- 双来源适配：同时支持 TG\+飞书双平台自动收集内容（auto 类型）和人工手写笔记（manual 类型），采用同一套加工规则，无需分开配置

- 多模型兼容：支持多种 LLM 模型，可通过配置文件修改 API 地址、密钥和模型名称，快速切换模型，适配不同需求（如成本控制、性能需求）

- 重试机制：内置重试逻辑，当 LLM API 调用失败时，自动重试指定次数，确保加工流程不中断，失败时保留原始内容，避免数据丢失

- 标准化输出：所有加工后的内容，均遵循统一的格式规范，确保结构化笔记的一致性，便于后续索引和知识网络构建

# 5\. 文件监控模块设计

## 5\.1 核心定位

文件监控模块是实现人工笔记\&\#34;保存即自动整理\&\#34;的关键，采用轻量化设计、无第三方依赖，以后台异步线程形式运行，不阻塞主流程，实时监控人工手写笔记目录的变化，触发自动加工流程。

## 5\.2 核心功能

- 实时监控：持续监控 raw/manual/ 目录，捕捉新文件创建、现有文件修改两种事件，确保所有手写笔记都能被及时检测

- 异步处理：以后台线程形式运行，与 NoteAgents 主流程（如 TG/飞书机器人、任务调度）完全隔离，不占用主流程资源，不影响核心功能运行

- 自动触发加工：检测到目录变化后，自动读取文件内容，提交至 LLM 处理模块进行结构化加工，无需人工手动触发

- 异常处理：遇到文件读取失败、加工失败等异常情况时，不中断监控流程，跳过异常文件，继续监控其他内容，同时记录异常信息至操作日志，便于后续排查

- 可配置性：监控间隔可通过配置文件调整，适配不同用户的使用习惯，平衡监控灵敏度和资源占用

## 5\.3 运行机制

模块启动后，自动扫描 raw/manual/ 目录下已有的所有 \.md 文件，记录为已监控文件；之后按照设定的间隔，定期扫描目录，对比当前文件列表与已监控文件列表，识别新增文件和修改文件；对新增和修改的文件，逐一进行读取和加工处理，处理完成后更新已监控文件列表，循环执行监控逻辑。

# 6\. Wiki 知识网络（Karpathy 范式）

## 6\.1 核心定位

Wiki 知识网络是知识深度沉淀的核心，贴合 Karpathy 标准范式，基于结构化内容，提炼实体、概念等核心知识，构建相互关联的知识体系，实现知识的复用、检索和深度关联，让碎片化内容形成完整的知识网络。

## 6\.2 核心操作

Wiki 知识网络的核心操作分为三类，覆盖知识的导入、检索和维护，确保知识网络的完整性和可用性：

1. **导入（Ingest）**: 将结构化内容导入 Wiki 知识网络，自动提取其中的实体、概念，创建对应的标准化页面，同时建立页面间的双向链接，构建知识关联

2. **查询（Query）**: 基于全局索引，检索 Wiki 知识网络中的相关页面，通过 LLM 综合所有相关内容，生成简洁、准确的查询结果，实现知识的快速复用

3. **健康检查（Lint）**: 定期对 Wiki 知识网络进行健康检查，识别孤立页面（无任何双向链接）、断裂链接（链接页面不存在）、过时内容等问题，标记需要优化的内容，保障知识网络的可用性

## 6\.3 标准化页面规范

Wiki 知识网络中的页面均采用标准化模板，确保格式统一、信息完整，便于关联和检索，核心页面类型包括：

1. **实体页**: 用于记录人、组织、产品等实体信息，包含一句话总结、详细信息、相关概念、来源等核心内容，明确实体的核心属性和关联知识

2. **概念页**: 用于记录技术、认知等概念信息，包含一句话总结、详细说明、相关实体、来源等核心内容，明确概念的定义、应用场景和关联知识

## 6\.4 全局索引规范

全局索引（index\.md）是 Wiki 知识网络的导航核心，自动更新，包含以下核心内容：

- 更新时间：显示索引的最后更新时间，确保用户了解索引的时效性

- 导航目录：按层级展示原始存档、结构化笔记、Wiki 知识网络的所有子目录链接，便于快速定位内容

- 知识库统计：展示原始存档、结构化笔记、实体、概念等各类内容的数量统计，让用户直观了解知识库规模

在 NoteAgents 现有架构基础上，新增 LLM Wiki 模块，与现有笔记收集、工具集模块并行，通过任务调度后置回调实现流程衔接，不破坏现有核心架构，具体扩展逻辑如下：

3. 交互入口层保持不变，系统已支持的 TG Bot、Feishu Bot 收集的内容，通过现有感知模块、决策模块、任务调度模块，传递至笔记收集模块，适配双平台自动收集需求

4. 新增 LLM Wiki 模块，包含内容加工、文件监控、索引管理等核心能力，与现有笔记收集模块、工具集模块并行，接受任务调度模块的调度

5. 通过任务调度后置回调机制，笔记收集模块收集的内容，自动触发 LLM Wiki 模块的加工流程，实现\&\#34;收集即加工\&\#34;；人工手写笔记通过文件监控模块触发加工流程，实现\&\#34;保存即加工\&\#34;

6. LLM Wiki 模块加工完成后，自动更新全局索引，同时将结构化内容同步至 Obsidian 知识库，实现与现有工具集的无缝衔接

- 笔记处理：识别新笔记（TG/飞书双平台收集、人工手写）的处理意图，调度 LLM 加工功能，完成结构化处理

为便于用户操作 LLM Wiki 功能，设计一套简洁、易用的命令，适配 NoteAgents 现有命令体系，核心命令包括：

- Wiki 查询命令：通过指定问题，查询 Wiki 知识库中的相关内容，返回综合整理后的结果

- 内容导入命令：手动输入内容，导入至 Wiki 知识库，触发 LLM 加工和知识沉淀

- 健康检查命令：手动触发 Wiki 知识网络健康检查，查看孤立页面、断裂链接等问题

- 索引查看命令：查看全局索引，快速导航至各类内容

- 最近笔记命令：查看近期加工的结构化笔记，了解知识库最新更新内容

# 8\. 配置设计

## 8\.1 核心配置维度

为实现 LLM Wiki 功能的灵活适配，设计标准化配置体系，涵盖功能开关、LLM 配置、路径配置、处理配置、Wiki 配置五个核心维度，所有配置可通过配置文件或环境变量修改，无需修改核心逻辑。

## 8\.2 核心配置内容

1. **功能开关**: 控制 LLM Wiki 功能的启用/禁用，便于根据需求灵活开启或关闭该功能

2. **LLM 配置**: 配置 LLM 模型的 API 地址、API 密钥、模型名称、重试次数，支持多种模型切换，同时适配不同模型的 API 格式

3. **路径配置**: 配置 Wiki 根目录、原始存档目录（自动/人工）、结构化内容目录的路径，可根据 NoteAgents 现有存储结构灵活调整；同时支持对 structured/ 目录下的子目录进行新增、删除配置，以及内容迁移相关路径适配

4. **处理配置**: 控制是否自动进行 LLM 精加工、是否监控人工手写笔记、监控间隔（秒），适配不同用户的使用习惯和性能需求

5. **Wiki 配置**: 控制是否自动导入结构化内容至 Wiki 知识网络、健康检查周期（天），保障 Wiki 知识网络的及时性和可用性

## 8\.3 环境变量配置

为便于部署和环境隔离，核心配置支持通过环境变量设置，涵盖功能开关、LLM API 配置、路径配置、处理配置等核心内容，可根据部署环境（开发、测试、生产）灵活调整，同时规避 API 密钥等敏感信息硬编码的风险。

3. **Phase 3（索引管理）**: 实现索引管理功能，完成全局索引的自动更新、内容统计等核心能力，确保索引与实际内容一致

4. **Phase 4（文件监控）**: 实现文件监控模块的核心功能，完成人工手写笔记的自动检测和加工触发，实现\&\#34;保存即整理\&\#34;

5. **Phase 5（Wiki 知识网络）**: 实现 Wiki 知识网络的导入、查询、健康检查等核心功能，完成标准化页面构建和知识关联

按照\&\#34;核心功能优先、增强功能次之、集成功能最后\&\#34;的原则，划分开发优先级，确保核心功能快速落地，满足基本使用需求，具体优先级如下：

1. **P0 \- 核心功能**: Wiki 目录结构创建、LLM 内容加工、分类保存、全局索引更新，是 LLM Wiki 功能的基础，确保内容能正常加工和沉淀

2. **P1 \- 增强功能**: 文件监控、Wiki 知识网络（导入、查询），提升功能易用性和知识沉淀能力，实现\&\#34;全自动整理\&\#34;和\&\#34;知识关联\&\#34;

# 10\. 与纯 Karpathy 方案对比

本方案基于 Karpathy 标准范式优化而来，保留其\&\#34;轻量、文件优先\&\#34;的核心思想，同时针对实际使用场景进行优化，解决纯 Karpathy 方案的痛点，具体对比如下：

- **内容来源**: 纯 Karpathy 方案仅支持手动导入内容，本方案支持 TG\+飞书双平台自动收集 \+ 人工手写双来源，适配更多使用场景

- **处理方式**: 纯 Karpathy 方案需手动触发内容加工，本方案采用后台异步自动处理，实现\&\#34;收集即加工、保存即整理\&\#34;，提升效率

- **分类体系**: 纯 Karpathy 方案无固定分类体系，内容管理混乱，本方案设计 tech/idea/study/daily 固定分类，实现内容标准化管理

- **索引功能**: 纯 Karpathy 方案仅提供基础 index\.md 索引，本方案实现全局索引自动更新，同时包含分类导航和内容统计，提升检索效率

- **文件监控**: 纯 Karpathy 方案无文件监控功能，本方案内置轻量化文件监控，实现人工笔记自动处理，降低使用门槛

- **基础设施**: 纯 Karpathy 方案仅依赖文件和模型，本方案新增轻量化自研模块，实现全自动化处理，同时保持无框架、轻量的特点

# 11\. 总结

本设计文档基于 NoteAgents 现有架构，整合 Karpathy 标准范式与豆包优化思路，设计的 LLM Wiki 功能，核心实现了双内容来源统一加工、三层目录分层隔离、全自动化处理，同时保持纯 Python 自研、无框架的特点，适配 NoteAgents 项目需求。

核心价值如下：

1. 知识只编译一次，后续可直接复用，避免重复加工，提升知识利用效率

2. 保存即自动整理，无需人工手动触发加工流程，降低使用门槛，提升使用体验

3. 原始文件永久保留，可追溯、可复用，保障数据安全，同时支持二次加工

4. 构建相互关联的知识网络，将碎片化内容沉淀为系统化知识，提升知识沉淀能力

5. 多模型兼容、可配置性强，适配不同部署环境和使用需求，便于后续扩展和维护

需注意，文档中初始配置的通义千问兼容模式 API 链接存在解析失败问题，后续实施时需替换为可正常访问的 LLM API 地址，或检查 API 权限与格式配置，确保 LLM 加工功能正常运行。
